Hackathon Project 2: Intelligent Next Best Action Platform

Flow360 Workforce Next Best Action Platform

A reusable agentic decision intelligence platform for B2B staffing operations. It turns interactions, enterprise knowledge, and persistent memory into explainable next best actions with human review.

Planner-led agents Supabase memory LlamaIndex retrieval Groq reasoning Human approval

Initial Problem

Build decision infrastructure, not another chatbot.

The hackathon asked for an agentic platform that can transform customer interactions and enterprise knowledge into actionable, explainable recommendations. The judging emphasis is on reusable architecture, memory, orchestration, business reasoning, and a polished user experience.

Business challenge

Business users receive meeting notes, emails, CRM updates, and policy information in scattered systems. They need to decide what to do next quickly, but the context required for that decision lives across documents, history, and people knowledge.

Technical challenge

A simple RAG answer is not enough. The system needs a planner, specialist agents, retrieval, reasoning, evidence, confidence, review, and memory updates that improve future recommendations.

Demo challenge

The product must tell a clear story in five minutes: ingest context, run the planner, show ranked actions, explain evidence, accept or reject, and update memory.

Chosen Solution

Flow360 focuses on workforce and staffing intelligence.

The selected B2B domain is healthcare and IT staffing operations. The demo account is Aarogya Health Network, a fictional strategic client with urgent ICU nurse and Epic analyst roles, credentialing deadlines, CFO approval friction, and renewal risk.

What the platform does

Flow360 ingests customer interactions and enterprise documents, retrieves account and policy context, analyzes risk and opportunity, generates ranked next best actions, explains why each action matters, and stores human feedback as memory.

Why this use case works

Staffing decisions are urgent, context-heavy, and business-critical. A missed credentialing detail can cause an SLA breach. A delayed CFO approval can lose a candidate. A poor stakeholder update can increase renewal risk.

Business process: staffing fulfillment User: account manager or recruiter Decision point: what should happen next Success: faster approvals, lower SLA risk

End-to-End Workflow

One operating loop covers the complete demo.

The flow is intentionally simple enough to explain, but agentic enough to show real orchestration and extensibility.

01

Ingest

Capture transcripts, emails, CRM notes, playbooks, PDFs, and uploaded files.

02

Retrieve

Search semantic chunks, account memory, candidate data, rules, and history.

03

Analyze

Identify urgency, risks, missing information, opportunities, and decision points.

04

Recommend

Create ranked next best actions with owner, due date, confidence, and evidence.

05

Review

Human approval or rejection updates persistent episodic memory.

Architecture

Planner orchestration with reusable specialist agents.

LangGraph defines the workflow as explicit nodes. The implementation also includes a fallback sequential path so the demo can still run if LangGraph is unavailable.

Frontend Next.js App Router, TypeScript, Tailwind, dashboard, memory graph, evidence panels.
API Layer FastAPI endpoints for ingest, planner runs, recommendations, review, and memory query.
Agent Workflow Planner, ingestion, retrieval, analyst, recommendation, and memory nodes.
Memory Store Supabase Postgres, pgvector chunks, recommendation feedback, agent run history.

LLM reasoning

Groq is used for fast reasoning and structured JSON generation. API keys are read from environment variables and rotated by the backend router. No keys should be committed to source control.

Retrieval

LlamaIndex-style chunking prepares documents for retrieval. Supabase pgvector is used for similarity search when available, with a keyword fallback for demo safety.

Embeddings

Ollama nomic-embed-text:latest creates local embeddings, which are stored on document chunks and used for context retrieval.

Agent Responsibilities

Each agent has a narrow job.

Planner Agent

Selects the workflow and decides the order of specialist agents for the staffing situation.

Ingestion Agent

Stores interactions as raw memory and chunks them into semantic memory for retrieval.

Retrieval Agent

Gathers account history, candidate context, policies, playbooks, and prior decisions.

Business Analyst Agent

Finds risks, opportunities, urgency, missing information, and decision points.

Recommendation Agent

Creates ranked next best actions with confidence, rationale, evidence, owner, and due date.

Memory Agent

Prepares and writes memory updates after human review so the system improves over time.

Memory Design

The memory layer is more than naive RAG.

Flow360 separates memory by purpose. This makes recommendations explainable and lets future agent runs use both documents and prior human decisions.

Raw memory

Original transcripts, emails, CRM notes, PDFs, and uploaded documents.

Semantic memory

Chunked and embedded document content for retrieval over enterprise knowledge.

Episodic memory

Agent runs, accepted or rejected recommendations, and reviewer notes.

Profile memory

Account-level summaries such as stakeholder map, renewal sensitivity, and known blockers.

Rule memory

Playbooks, credentialing policies, SLA rules, escalation policies, and commercial thresholds.

Ask Memory

The dashboard includes a memory question panel that answers using persistent memory and retrieved evidence.

How A Planner Run Works

The full runtime sequence.

User action: the account manager enters or accepts the demo interaction and clicks Run Planner. The frontend calls POST /agent/run.

  1. The backend creates a run id and initializes state with account id, objective, and interaction.
  2. The planner node writes a five-step staffing intelligence plan.
  3. The ingestion node stores the interaction as a document and creates semantic chunks.
  4. The retrieval node searches Supabase pgvector or fallback keyword search for relevant evidence.
  5. The analyst node asks the reasoning model for structured JSON analysis.
  6. The recommendation node asks the reasoning model for ranked next best actions.
  7. The memory node prepares profile and episodic memory updates.
  8. The API saves the run trace and recommendations, then returns them to the dashboard.
  9. The user approves or rejects a recommendation through POST /recommendations/{id}/review.
  10. The review writes an episodic memory card so later decisions know what happened.

API Surface

Backend endpoints implemented.

Endpoint Purpose Used by
GET /health Checks backend, Groq, and Supabase status. Developer verification
GET /demo/state Returns account, metrics, recommendations, memory cards, and demo interaction. Dashboard initial state
POST /ingest/text Stores pasted text as a document and semantic chunks. Backend ingestion
POST /ingest/upload Extracts text from uploaded files and indexes it. Intake view
POST /agent/run Runs the planner and specialist agents end to end. Run Planner button
GET /agent/runs/{run_id} Fetches a saved agent run. Trace and auditability
GET /recommendations Lists saved recommendations for an account. Recommendation inbox
POST /recommendations/{id}/review Approves or rejects an action and stores feedback memory. Human-in-the-loop review
GET /memory/{entity_type}/{entity_id} Returns persistent memory cards. Memory page
POST /memory/query Answers questions over persistent memory and retrieved evidence. Ask Memory panel

Frontend Experience

A decision workspace, not a chat screen.

Landing page

The landing page is a coded premium AI SaaS hero inspired by the reference visual: dark cinematic frame, strong side vignettes, large Flow360 backdrop text, centered headline, and embedded dashboard preview. It avoids using the reference screenshot as the actual page background.

Operator console

The dashboard minimizes clicks: recommendation inbox, selected action details, evidence, agent trace, upload/intake, memory graph, and Ask Memory are reachable from compact left navigation.

Memory graph UI

Memory is represented visually as account, profile, rule, episodic, semantic, and raw memory nodes. This communicates that the platform remembers business context over time.

Human review

The selected action panel lets the user approve or reject recommendations. The UI then updates status and adds reviewer feedback into memory.

Project Files

Where the main work lives.

Frontend

frontend/src/app/page.tsx contains the landing page. frontend/src/components/flow-dashboard.tsx contains the operator console, recommendation workflow, memory graph, intake, trace, and Ask Memory UI.

Backend

backend/app/main.py exposes the FastAPI routes. backend/app/agents/workflow.py defines the LangGraph planner workflow. backend/app/services/store.py handles Supabase and fallback memory.

Data and SQL

data/mock_docs/ stores mock enterprise documents. supabase/sql/ contains database scripts for tables, pgvector, and retrieval RPCs.

Documentation

docs/setup.md, docs/demo_script.md, and docs/architecture_walkthrough.md support setup, demo delivery, and architecture explanation.

Security note

API keys and Supabase secrets belong only in local .env files. They should not appear in source code or in this explainer. Any keys shared during development should be regenerated before submitting the GitHub repository.

Why It Should Impress

The project maps directly to the judging criteria.

Platform quality

The system has explicit orchestration, reusable agents, persistent memory, retrieval, evidence, confidence, and human review. These are platform capabilities, not isolated chatbot features.

Business reasoning

The recommendations tie staffing operations to measurable outcomes: SLA breach risk, start date protection, approval speed, renewal confidence, and stakeholder trust.

Extensibility

New domains can reuse the same loop by changing documents, rules, prompts, memory schemas, and specialist agents. Customer success, SaaS sales, and field service could follow the same architecture.

Demo strength

The demo can be completed in under five minutes: open dashboard, run planner, inspect recommendation evidence, review an action, and show memory updated.
